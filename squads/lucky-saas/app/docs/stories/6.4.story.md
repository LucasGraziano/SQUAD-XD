## User Story

Como produto, preciso que o status de assinatura de cada corretor seja atualizado automaticamente quando Stripe processa pagamentos, upgrades, downgrades ou cancelamentos — para que o acesso às features seja sempre consistente com o plano pago.

---

## Value Proposition

- Sem webhooks, o sistema não sabe quando alguém pagou, cancelou ou ficou inadimplente
- Sem Billing Portal, o corretor não consegue trocar cartão ou ver faturas (causa churn evitável)
- Implementa o lifecycle completo definido no `billing-spec.md`

---

## Acceptance Criteria

- [ ] **AC-1:** Endpoint `POST /api/webhooks/stripe` processa eventos Stripe com verificação de assinatura (`stripe.webhooks.constructEvent`)
- [ ] **AC-2:** `customer.subscription.created` → cria/atualiza registro em `brokers` com `plan`, `subscription_id`, `subscription_status: 'active'`, `current_period_end`
- [ ] **AC-3:** `customer.subscription.updated` → atualiza `plan` e `subscription_status` em tempo real (cobre upgrade e downgrade)
- [ ] **AC-4:** `customer.subscription.deleted` → marca `subscription_status: 'canceled'`, inicia grace period de 30 dias
- [ ] **AC-5:** `invoice.payment_succeeded` → confirma pagamento, renova `current_period_end`
- [ ] **AC-6:** `invoice.payment_failed` → marca `subscription_status: 'past_due'`, envia e-mail de cobrança via Resend: "Seu pagamento falhou — atualize seu cartão para manter o acesso"
- [ ] **AC-7:** `customer.subscription.trial_will_end` (3 dias antes) → envia e-mail via Resend: "Seu trial encerra em X dias — assine para não perder seus dados"
- [ ] **AC-8:** Link "Gerenciar assinatura" em `/configuracoes` → abre Stripe Customer Portal (trocar cartão, ver faturas, cancelar)
- [ ] **AC-9:** Trial expirado sem conversão (após `trial_ends_at`) → middleware bloqueia todas as rotas do dashboard exceto `/pricing` e `/account`
- [ ] **AC-10:** Função `getBrokerPlan(userId)` retorna `{ plan, status, trialEndsAt, periodEnd }` — usada pelo middleware e pelos Server Components para verificar acesso

## Technical Spec

### Webhook Handler

```typescript
// src/app/api/webhooks/stripe/route.ts
export async function POST(req: Request) {
  const sig = req.headers.get('stripe-signature')!;
  const body = await req.text();
  const event = stripe.webhooks.constructEvent(body, sig, process.env.STRIPE_WEBHOOK_SECRET!);

  switch (event.type) {
    case 'customer.subscription.created':
    case 'customer.subscription.updated':
      await syncSubscription(event.data.object as Stripe.Subscription);
      break;
    case 'customer.subscription.deleted':
      await handleCancellation(event.data.object as Stripe.Subscription);
      break;
    case 'invoice.payment_failed':
      await handlePaymentFailed(event.data.object as Stripe.Invoice);
      break;
    case 'customer.subscription.trial_will_end':
      await sendTrialEndingEmail(event.data.object as Stripe.Subscription);
      break;
  }
  return Response.json({ received: true });
}
```

export async function getBrokerPlan(userId: string) {
  const { data } = await supabase
    .from('brokers')
    .select('plan, subscription_status, trial_ends_at, current_period_end')
    .eq('user_id', userId)
    .single();
  return data;
}

## Scope

**IN:**
- Webhook handler para 5 eventos críticos (billing-spec.md)
- Stripe Customer Portal para autogestão
- Middleware de bloqueio por status de assinatura
- E-mails transacionais: pagamento falhou, trial encerrando

**OUT:**
- Retry manual de pagamento via UI (Stripe Portal cobre isso)
- Relatório de receita (Stripe Dashboard)
- Invoice customizada com logo do produto
- Downgrade logic complexa (dados em read-only — já especificado em billing-spec.md; middleware implementa o acesso)

## Risks

- Raw body obrigatório para verificação de assinatura — Next.js App Router requer `await req.text()` (não `req.json()`)
- Downgrade mid-cycle: Stripe aplica no próximo ciclo — middleware deve verificar `current_period_end` não `status` para acesso durante o período pago
- `idempotencyKey` nos updates do banco — webhooks Stripe podem chegar duplicados
