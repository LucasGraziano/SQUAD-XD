## User Story

Como corretor usando o Premia no celular, quero receber notificações push quando um cliente tiver uma apólice vencendo ou uma pendência atrasada, para ser alertado mesmo sem o app aberto.

---

## Value Proposition

- Corretor brasileiro usa 80% mobile — notificação push é o canal mais imediato
- PWA elimina barreira de instalação (não precisa de App Store)
- Diferencial vs. BrokerOne e Quiver que são web-only sem push
- Aumenta DAU/MAU ao trazer corretores de volta ao produto diariamente (O2/KR2)

---

## Acceptance Criteria

- [ ] **AC-1:** `manifest.json` configurado com `name`, `short_name`, `icons` (192x192, 512x512), `display: "standalone"`, `theme_color`, `background_color`
- [ ] **AC-2:** Service Worker registrado — cache de assets estáticos com estratégia `cache-first`
- [ ] **AC-3:** Prompt de permissão de notificação exibido após o corretor configurar o primeiro alerta (não na abertura — momento certo = após valor demonstrado)
- [ ] **AC-4:** Push notification enviada 24h antes do vencimento de apólice com título: "⚡ [Nome do Cliente] — apólice vence amanhã" e body: "[Ramo] · [Seguradora]"
- [ ] **AC-5:** Push notification enviada quando pendência atinge prazo sem resolução: "⚠️ Pendência vencida: [Título]"
- [ ] **AC-6:** Clicar na notificação abre o app diretamente na apólice ou pendência correspondente (deep link via `data.url` no payload)
- [ ] **AC-7:** Corretor pode desativar push notifications em `/configuracoes` — toggle "Notificações push"
- [ ] **AC-8:** Subscriptions armazenadas em `push_subscriptions` com `broker_id` + endpoint + keys
- [ ] **AC-9:** Botão "Adicionar à tela inicial" sugerido após 3 sessões (não intrusivo — banner dismissível)

## Technical Spec

### Database

```sql
CREATE TABLE push_subscriptions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  broker_id UUID NOT NULL REFERENCES brokers(id) ON DELETE CASCADE,
  endpoint TEXT NOT NULL,
  p256dh TEXT NOT NULL,
  auth TEXT NOT NULL,
  created_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE (broker_id, endpoint)
);
ALTER TABLE push_subscriptions ENABLE ROW LEVEL SECURITY;
CREATE POLICY "broker own subscriptions" ON push_subscriptions
  FOR ALL USING (broker_id = (SELECT id FROM brokers WHERE user_id = auth.uid()));
```

### Web Push (web-push package)

```typescript
// src/lib/push.ts
import webpush from 'web-push';
webpush.setVapidDetails('mailto:...', VAPID_PUBLIC, VAPID_PRIVATE);

export async function sendPushNotification(subscription: PushSubscription, payload: PushPayload) {
  await webpush.sendNotification(subscription, JSON.stringify(payload));
}
```

### Cron Job — Alertas Push

```typescript
// src/app/api/cron/push-alerts/route.ts
// Disparado diariamente às 8h pelo Vercel Cron
// Busca apólices com vencimento em 24h → envia push para cada broker_id com subscription ativa
// Busca pendências com due_date = hoje → envia push de pendência vencida
```

```
NEXT_PUBLIC_VAPID_PUBLIC_KEY=...

```

## Scope

**IN:** PWA manifest, Service Worker, push permissions, alertas de vencimento e pendência, deep link  
**OUT:** Push para e-mail automático enviado (muito frequente), push marketing/promoções, iOS Safari (limitações de push — informar usuário)

---

## Risks

- iOS Safari: push notifications via Web Push API ainda com suporte limitado (iOS 16.4+, apenas via PWA instalado) — documentar limitação para usuário e testar em iPhone real
- VAPID keys: se rotacionadas em produção, todas as subscriptions existentes invalidam — backup das keys obrigatório
- `next-pwa` pode conflitar com App Router do Next.js 15 — verificar compatibilidade antes de instalar; alternativa: service worker manual

---

## Dependencies

- `npm install web-push next-pwa`
- Vercel Cron configurado (ou adaptar cron existente de alertas)
- VAPID keys geradas
