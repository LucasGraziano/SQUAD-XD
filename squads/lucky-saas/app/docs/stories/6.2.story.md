## User Story

Como corretor usando o Premia, quando tento acessar uma feature além do meu plano atual, quero ver claramente o valor que estou perdendo e o custo do upgrade — para tomar uma decisão informada em vez de me sentir bloqueado arbitrariamente.

---

## Value Proposition

- Upsell no momento da dor tem conversão 3–5× maior que e-mail de marketing
- Uma implementação, todos os gates — elimina código duplicado de bloqueio por feature
- Calculadora de ROI embutida transforma objeção de preço em decisão racional
- Linguagem de progresso (não de punição) reduz churn causado por frustração com paywall

---

## Acceptance Criteria

- [ ] **AC-1:** Componente `<PlanGate requiredPlan="profissional" feature="portal-cliente" modal="soft">` implementado e reutilizável — envolve qualquer elemento filho, intercepta clique/acesso e exibe modal se o plano atual for insuficiente
- [ ] **AC-2:** Dois modos de modal:
  - `modal="soft"` — exibe overlay com opção de fechar; navegação mantida
  - `modal="hard"` — bloqueia completamente a UI (usar apenas para limites de dados: 100 clientes, 2º usuário)
- [ ] **AC-3:** Os 5 triggers do PRD implementados com copy correto:

  | Trigger | Modal | Copy âncora |
  |---------|-------|-------------|
  | Tentativa de 2º usuário | hard | "Trabalho em equipe requer o plano Equipe" |
  | 100º cliente atingido | hard | "Você chegou ao limite — continue crescendo" |
  | Portal do Cliente clicado (Solo) | soft | "Uma renovação salva paga 12 meses de assinatura" |
  | E-mail automático bloqueado (Solo) | soft | "ROI calculado: X renovações × comissão" |
  | WhatsApp API na sidebar (Pro) | soft | Demo + calculadora de ROI |

- [ ] **AC-4:** Modal exibe preço do plano de destino + CTA "Fazer upgrade" → redireciona para `/pricing?highlight={plan}`
- [ ] **AC-5:** Calculadora de ROI embutida no modal de e-mail automático:
  - Inputs: carteira (slider 50–500), comissão média (campo, default R$450), renovações perdidas/mês (slider 1–10)
  - Output dinâmico: "Recuperar X renovações/mês = **Y× o plano Profissional**"
  - Cálculo: `(renovacoes × comissao) / 97` = multiplicador
- [ ] **AC-6:** Evento PostHog disparado em cada exibição: `plan_gate_shown` com propriedades `{ feature, current_plan, required_plan, modal_type }`
- [ ] **AC-7:** Evento PostHog disparado ao clicar em upgrade: `upgrade_clicked` com `{ from_plan, to_plan, trigger_feature }`
- [ ] **AC-8:** Princípios de copy aplicados em todos os modais:
  - ❌ Nunca: "Faça upgrade para desbloquear"
  - ✅ Sempre: linguagem de progresso + relativização de preço + framing de ROI
- [ ] **AC-9:** `<PlanGate>` integrado nos 5 pontos de trigger mapeados no PRD (substituindo qualquer lógica de bloqueio ad-hoc existente)
- [ ] **AC-10:** Componente é server-component-safe: lê plano do usuário via prop (`currentPlan`) — não faz fetch interno

## Technical Spec

### Componente Principal

```tsx
// src/components/shared/PlanGate.tsx
interface PlanGateProps {
  requiredPlan: 'profissional' | 'equipe' | 'enterprise';
  feature: 'portal-cliente' | 'email-automatico' | 'whatsapp-api' | 'multi-usuario' | 'limite-clientes';
  modal: 'soft' | 'hard';
  currentPlan: 'solo' | 'profissional' | 'equipe' | 'enterprise' | 'trial';
  children: React.ReactNode;
}

// Lógica: se currentPlan >= requiredPlan → render children normalmente
// Se não → intercepta clique e exibe modal correto
```

### Hierarquia de Planos

```typescript
const PLAN_HIERARCHY = { solo: 0, profissional: 1, equipe: 2, enterprise: 3 };
// trial = profissional para verificação de gates
const isAllowed = (current: Plan, required: Plan) =>
  PLAN_HIERARCHY[current === 'trial' ? 'profissional' : current] >= PLAN_HIERARCHY[required];
```

## Scope

**IN:**
- Componente `<PlanGate>` genérico e reutilizável
- 5 triggers mapeados no PRD implementados
- Calculadora de ROI no modal de e-mail automático
- Analytics PostHog para os 2 eventos P0
- Substituição de qualquer lógica ad-hoc de bloqueio existente

**OUT:**
- Integração com Stripe para upgrade direto no modal (redireciona para `/pricing`)
- A/B testing de copy dos modais (fase futura via PostHog feature flags)
- Animação de demo do WhatsApp (AC-3 trigger 5 — apenas text + CTA nesta story)
- NPS pós-upgrade

## Risks

- Gates implementados parcialmente no código atual (verificações ad-hoc) — é preciso auditar todos os pontos e substituir pelo componente unificado para evitar bypass acidental
- currentPlan precisa ser passado corretamente via Server Component → Client Component; não usar fetch no lado do cliente para evitar flash de conteúdo bloqueado
