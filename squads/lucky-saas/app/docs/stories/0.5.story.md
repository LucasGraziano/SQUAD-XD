# Story 0.5 — Billing Management: Upgrade e Cancelamento de Plano

## User Story

Como corretor, quero poder fazer upgrade do meu plano ou cancelar minha assinatura diretamente pelo painel — sem precisar mandar e-mail ou falar com suporte.

---

## Value Proposition

- Self-serve billing = menos fricção no upgrade → mais conversão
- Cancel sem suporte = confiança (o corretor sabe que não vai ficar preso)
- Trial que cancela na hora = respeito ao usuário
- Billing Portal do Stripe = zero UI de pagamento pra manter

---

## Acceptance Criteria

- [ ] **AC-1:** Botão "Fazer upgrade" na seção de Plano em `/configuracoes` leva o usuário ao Stripe Checkout para o plano desejado (Solo → Pro, Solo → Equipe, Pro → Equipe)
- [ ] **AC-2:** Usuário em trial ativo vê botão "Cancelar trial" — ao confirmar, o trial é encerrado imediatamente e o plano retorna a inativo (sem cobrança)
- [ ] **AC-3:** Usuário com assinatura ativa vê botão "Cancelar assinatura" — ao confirmar, `cancel_at_period_end = true` no Stripe; assinatura continua até o fim do período pago; badge mostra "Cancela em [data]"
- [ ] **AC-4:** Webhook do Stripe atualiza `subscription_status`, `plan` e `trial_ends_at` na tabela `brokers` automaticamente ao receber eventos de lifecycle
- [ ] **AC-5:** Usuário com plano ativo (não Solo) vê botão "Gerenciar assinatura" que abre o Stripe Billing Portal (troca de cartão, histórico de faturas, upgrade/downgrade)
- [ ] **AC-6:** Modal de confirmação antes de qualquer ação de cancelamento — sem cancelamento acidental
- [ ] **AC-7:** Após upgrade bem-sucedido, plano atualiza em tempo real na interface (revalidação do servidor)
- [ ] **AC-8:** Estados de erro tratados com mensagem clara (Stripe indisponível, cartão recusado, etc.)

---

## Comportamento por Situação

| Situação | Botão exibido | Ação |
|----------|---------------|------|
| Solo (sem assinatura) | "Fazer upgrade → Pro" + "Fazer upgrade → Equipe" | Stripe Checkout |
| Trial ativo | "Cancelar trial" | Cancela imediatamente no DB |
| Assinatura ativa | "Gerenciar assinatura" + "Cancelar assinatura" | Billing Portal / cancel_at_period_end |
| Cancelamento agendado | Badge "Cancela em [data]" + "Reativar assinatura" | Remove cancel_at_period_end |
| Plano Pro | "Fazer upgrade → Equipe" + "Gerenciar assinatura" | Checkout / Billing Portal |

## Technical Spec

### Stack
- Stripe SDK (`stripe` npm package) — server-side only
- `STRIPE_SECRET_KEY` — obrigatório
- `STRIPE_WEBHOOK_SECRET` — obrigatório para AC-4
- `STRIPE_PRO_PRICE_ID` — ID do preço Pro no Stripe
- `STRIPE_BROKER_PRICE_ID` — ID do preço Equipe no Stripe
- `NEXT_PUBLIC_APP_URL` — para redirect após checkout

### Server Actions (`src/app/actions/billing.ts`)

```typescript
// Cria sessão de Checkout para upgrade
createCheckoutSession(targetPlan: 'pro' | 'broker'): Promise<{ url: string | null; error: string | null }>

// Cria sessão do Billing Portal (gerenciar assinatura)
createBillingPortalSession(): Promise<{ url: string | null; error: string | null }>

// Cancela trial imediatamente (sem Stripe — só DB)
cancelTrial(): Promise<{ error: string | null }>

// Cancela assinatura ao fim do período (cancel_at_period_end = true)
cancelSubscription(): Promise<{ error: string | null }>

// Reativa assinatura cancelada (remove cancel_at_period_end)
reactivateSubscription(): Promise<{ error: string | null }>
```

### Webhook (`src/app/api/webhooks/stripe/route.ts`)

Eventos a tratar:
- `checkout.session.completed` → atualiza plan + subscription_status = 'active'
- `customer.subscription.updated` → sincroniza plan, status, cancel_at_period_end
- `customer.subscription.deleted` → plan = 'starter', status = 'canceled'
- `invoice.payment_failed` → status = 'past_due'

### Schema DB (brokers)
Campos existentes que serão usados:
- `plan` — 'starter' | 'pro' | 'broker'
- `subscription_status` — 'trialing' | 'active' | 'canceled' | 'past_due' | null
- `trial_ends_at` — TIMESTAMPTZ

Campos a adicionar via migration:
- `stripe_customer_id` TEXT — ID do cliente no Stripe
- `stripe_subscription_id` TEXT — ID da assinatura ativa
- `cancel_at_period_end` BOOLEAN DEFAULT false
- `current_period_end` TIMESTAMPTZ — quando a assinatura atual expira

### UI (`ConfiguracoesClient.tsx`)
- Seção "Plano e Billing" expandida com botões contextuais
- Modal de confirmação de cancelamento (estado local)
- Loading state durante redirect para Stripe
