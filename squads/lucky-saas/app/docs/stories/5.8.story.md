## Contexto

O portal do cliente atual mostra apenas informações básicas da apólice. Para o cliente se autoatender sem precisar falar com o corretor, o portal precisa expor: o bem segurado, as coberturas contratadas em linguagem humana, franquia, e os contatos da seguradora para acionamento de sinistro.

No modal de apólice, o corretor já pode cadastrar esses dados via campos de metadata por ramo. Esta story exibe essas informações de forma rica no portal.

---

## Acceptance Criteria

- [x] **AC-1:** Campos de sinistro adicionados ao modal de apólice para todos os ramos: Telefone 24h e WhatsApp da seguradora
- [x] **AC-2:** Campo "Coberturas principais" adicionado ao modal (texto livre, uma por linha ou separadas por vírgula)
- [x] **AC-3:** Cards de apólice no portal são expansíveis — clique em "Ver detalhes" revela seção completa
- [x] **AC-4:** Seção expandida mostra: bem segurado, placa (auto), franquia, coberturas como tags, contatos de sinistro
- [x] **AC-5:** Botões de sinistro são clicáveis: tel → tel: link, WhatsApp → wa.me link
- [x] **AC-6:** Botão "Solicitar Renovação" por apólice → cria alerta para o corretor via API pública
- [x] **AC-7:** Estado do botão de renovação: idle → loading → "Solicitado ✓" (não repete)
- [x] **AC-8:** Se não há contatos de sinistro cadastrados, exibe "Contate seu corretor" com link WhatsApp do corretor

## Technical Spec

### Modal de Apólice — novos campos em `metadata`
- `sinistro_tel` — telefone 24h da seguradora (todos os ramos)
- `sinistro_zap` — WhatsApp da seguradora (todos os ramos)
- `coberturas` — coberturas separadas por vírgula (todos os ramos)

### Portal — componente client
- `PortalPolicyCard` — 'use client', gerencia expand/collapse e renewal request state
- Portal page passa `token` + `brokerWhatsappUrl` para cada card

### Renewal Request API
- `POST /api/portal/renewal-request` — pública, valida token, cria alerta no broker
