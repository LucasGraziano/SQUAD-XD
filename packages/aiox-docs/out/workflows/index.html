Workflows & Tasks

O AIOX orquestra desenvolvimento através de 4 workflows primários compostos por 207 tasks estruturadas. Cada workflow define a sequência, agentes e gates de qualidade.

Quando usar cada workflow?

-
New story from epic→Story Development Cycle
-
QA found issues, need iteration→QA Loop
-
Complex feature needs spec→Spec Pipeline → then SDC
-
Joining existing project→Brownfield Discovery
-
Simple bug fix→SDC only (YOLO mode)

Story Development Cycle (SDC)

Full 4-phase workflow for all development work. Create → Validate → Implement → QA Review.

PRIMARY
4 fases|Usar quando: New story from epic, any development task
1
🌊@sm→create-next-story.md

Create story from PRD/epic

Output: story.md
↓
2
🎯@po→validate-next-story.md

10-point validation checklist

Output: GO/NO-GO verdict
↓
3
💻@dev→dev-develop-story.md

Implement with CodeRabbit self-healing

Output: Working code
↓
4
✅@qa→qa-gate.md

7-point quality gate

Output: PASS/FAIL verdict

QA Loop

Automated review-fix cycle. @qa reviews → verdict → @dev fixes → re-review (max 5 iterations).

ITERATIVE
3 fases|Usar quando: QA found issues, need iteration
1
✅@qa→qa-review

Review code quality and criteria

Output: APPROVE/REJECT/BLOCKED
↓
2
💻@dev→dev-apply-qa-fixes

Apply fixes from QA feedback

Output: Fixed code
↓
3
✅@qa→qa-re-review

Re-review after fixes (max 5x)

Output: Final verdict

Spec Pipeline

Transform informal requirements into executable spec. 6 phases with complexity-based phase skipping.

PRE-IMPLEMENTATION
6 fases|Usar quando: Complex feature needs spec before coding
1
📋@pm→spec-gather

Gather requirements (never skip)

Output: requirements.json
↓
2
📐@architect→spec-assess

Assess complexity (5 dimensions, score 1-25)

Output: complexity.json
↓
3
🔍@analyst→spec-research

Research dependencies (skip if SIMPLE)

Output: research.json
↓
4
📋@pm→spec-write

Write specification (never skip)

Output: spec.md
↓
5
✅@qa→spec-critique

Critique spec (APPROVED/NEEDS_REVISION/BLOCKED)

Output: critique.json
↓
6
📐@architect→spec-plan

Implementation plan (if APPROVED)

Output: implementation.yaml

Brownfield Discovery

10-phase technical debt assessment for existing codebases. Data collection → drafting → validation → finalization.

ASSESSMENT
10 fases|Usar quando: Joining existing project, legacy assessment
1
📐@architect→analyze-brownfield

System architecture analysis

Output: system-architecture.md
↓
2
🗄️@data-engineer→db-schema-audit

Database audit (if DB exists)

Output: SCHEMA.md + DB-AUDIT.md
↓
3
🎨@ux-design-expert→frontend-spec

Frontend specification

Output: frontend-spec.md
↓
4
📐@architect→tech-debt-draft

Technical debt draft

Output: technical-debt-DRAFT.md
↓
5
🗄️@data-engineer→db-review

Database specialist review

Output: db-specialist-review.md
↓
6
🎨@ux-design-expert→ux-review

UX specialist review

Output: ux-specialist-review.md
↓
7
✅@qa→qa-review

QA gate (APPROVED/NEEDS WORK)

Output: qa-review.md
↓
8
📐@architect→finalize

Final assessment document

Output: technical-debt-assessment.md
↓
9
🔍@analyst→executive-report

Executive summary report

Output: TECHNICAL-DEBT-REPORT.md
↓
10
📋@pm→epic-stories

Create epic and stories for fixes

Output: Epic + stories
